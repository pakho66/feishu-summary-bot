## 项目运行架构（当前）

### 整体数据流

```
飞书用户 → 飞书开放平台 → 阿里云域名(zhanbohao.top) → Cloudflare CDN(国内加速) → Worker核心 → 外部服务(Redis/DeepSeek/飞书API)
                                                                                                    ↓
                                                                              飞书API回复消息原路返回 → 用户收到
```

### 分层说明

#### 1. 域名注册层 — 阿里云万网
- 域名：`zhanbohao.top`（阿里云购买，¥10/年）
- DNS 托管到 Cloudflare（NS 指向 `brett.ns.cloudflare.com` + `jo.ns.cloudflare.com`）
- 作用：提供自定义域名，解决 `workers.dev` 在中国被墙的问题

#### 2. CDN 加速层 — Cloudflare CDN
- 自定义域名 `https://zhanbohao.top` 挂在 Cloudflare
- 国内访问走 CDN 边缘节点加速
- SSL/TLS 自动管理
- 作用：国内可访问 + HTTPS

#### 3. Worker 核心层 — Cloudflare Workers
- 370 行纯 JavaScript，零依赖，冷启动 <5ms
- 三个入口：
  - **Webhook 处理器**：接收飞书消息，用 `ctx.waitUntil()` 异步处理（飞书立即收到 200，不超时）
  - **Cron 定时触发**：每天 18:00（UTC 10:00）自动触发，生成日报推送到私聊
  - **/health 诊断**：监控接口，检查配置和 Redis 连接状态
- 密钥管理：8 个 Cloudflare Secrets（加密存储），wrangler deploy 不会覆盖

#### 4. 外部服务层

| 服务 | 作用 | 调用方式 | 费用 |
|------|------|---------|------|
| Upstash Redis | 存对话记录，按日期分 key，7天过期 | 纯 fetch REST API | 免费 |
| DeepSeek API | AI 生成总结（待办事项→主要讨论→关键决策） | 兼容 OpenAI 格式的 fetch | 按token计费（极低） |
| 飞书 Open API | 发送消息回复用户 | `receive_id_type` 区分群聊/私聊 | 免费 |

### 关键技术决策

1. **Cloudflare Workers 替代 Vercel**：Vercel 冷启动 5-8 秒，飞书只等 3 秒；Workers 冷启动 <5ms
2. **自定义域名替代 workers.dev**：workers.dev 在中国被墙，必须用自定义域名 + Cloudflare CDN 加速
3. **DeepSeek 替代 OpenAI**：DeepSeek 兼容 OpenAI 格式，国内访问无障碍
4. **Upstash Redis 替代内存数组**：Workers 无状态，需要外部持久化存储
5. **纯 fetch 零依赖**：Redis/飞书/DeepSeek 全部用原生 fetch，不装任何 npm 包

### 第十阶段：切换到 Cloudflare Workers

**做的事**：
1. 决定从 Vercel 切换到 Cloudflare Workers（冷启动 <5ms）
2. 重写代码为 Cloudflare Workers 格式：
   - `export default { fetch(request, env, ctx), scheduled(event, env, ctx) }`
   - 环境变量从 `process.env` → `env` 参数

4. 创建 Cloudflare 账号（免费，无需信用卡）

- 添加诊断字段确认 envKeys 正确传递 → 最终全部 ✅

### 第十九阶段：机器人不回复——Bug 连环修复

**Bug #1：事件类型不匹配**

**现象**：发消息给机器人，无回复
**排查**：
- `curl` 模拟 webhook 请求 → 正常响应
- `wrangler tail` 查看实时日志 → 发现飞书请求到达了，但消息被跳过
**根因**：代码写的是 `im.message.receive`，飞书实际发的是 `im.message.receive_v1`
**修复**：改为 `body.header?.event_type?.startsWith('im.message.receive')` 兼容两种格式
**部署**：`wrangler deploy`

---

**Bug #2：飞书 API 权限不足**

**现象**：修复 Bug #1 后仍然不回复
**排查**：
- 添加 `/debug/feishu` 调试端点，测试 Token 获取 → OK
- 代码改为回复所有消息（调试模式）→ 仍不回复
- `wrangler tail` 日志显示：飞书请求到达、消息解析成功、Token 获取成功
- 但 `sendToFeishu` 返回错误码 `99991672`
**根因**：错误码 `99991672` = 权限不足，缺少 `im:message:send` / `im:message` / `im:message:send_as_bot` 权限
**修复**：
- 指导用户去飞书开放平台 → 权限管理 → 搜索 `im:message`
- 添加并开通 `im:message` 和 `im:message:send_as_bot` 权限
- 重新创建版本并发布应用

---

**Bug #3：sendToFeishu 缺少 receive_id_type 参数**

**现象**：权限开通后，飞书 API 仍报错
**排查**：日志显示 `sendToFeishu` 返回错误
**根因**：飞书发送消息 API 需要 `?receive_id_type=open_id` 查询参数，代码中缺失
**修复**：
```javascript
// 修复前
fetch('https://open.feishu.cn/open-apis/im/v1/messages', ...)
// 修复后
fetch('https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=open_id', ...)
```

**结果**：三个 Bug 修复后，机器人终于能回复了！🎉

### 第二十阶段：群聊回复到私聊——Bug #4

**现象**：在群里 @ 机器人，机器人在私聊回复
**根因**：`sendToFeishu` 始终用发送者的 `open_id` 发消息，没有区分群聊和私聊
**修复**：
1. `processMessage` 中提取 `chat_id`（群聊有值，私聊为空）
2. 判断 `isGroup = !!chatId`
3. 群聊用 `chat_id` + `receive_id_type=chat_id` 回复到群里
4. 私聊用 `open_id` + `receive_id_type=open_id` 回复到私聊

```javascript
async function sendToFeishu(token, receiveId, text, isGroup = false) {
  const idType = isGroup ? 'chat_id' : 'open_id';
  const response = await fetch(
    `https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=${idType}`, ...
  );
}
```

**结果**：群聊 @ 机器人 → 群里回复；私聊机器人 → 私聊回复

## 两次复盘的核心教训

### 第一轮（部署阶段）的教训
1. **冷启动是 Serverless 在中国的致命问题**：飞书只等 3 秒
2. **workers.dev 在中国被墙**：必须用自定义域名
3. **Vercel Edge Functions 不稳定**：免费版有 bug
4. **Dashboard Text 变量会被 deploy 覆盖**：必须用 Secret

### 第二轮（上线调试阶段）的教训
1. **飞书事件类型带版本后缀**：`_v1` 不能少，用 `startsWith` 更安全
2. **添加事件订阅 ≠ 开通权限**：这是两个独立步骤
3. **飞书 API 必填参数不能漏**：`receive_id_type` 是必填的
4. **群聊和私聊的回复方式不同**：`chat_id` vs `open_id`
5. **wrangler tail 是最有效的调试工具**：日志直接暴露问题根因
6. **代码能跑 ≠ 功能正常**：必须端到端测试
